# Registry Standard

Conventions for first-class resource registries in this project. One page. Grounded in the shape of the registries we have, not speculation.

## When to create one

Open a registry when you have a designer-authored `Resource` type backed by `.tres` files under `data/tres/<type>/`. Skip registries for:

- Runtime-only types (no `.tres` backing).
- Resources embedded inside other resources (e.g. `SkillLevelData` inside `SkillData`).
- Small closed sets that belong in constants or enums.

If a type meets the bar, the registry is the single source of truth for "does this X exist" and "what are all the Xs". Don't reconstruct that answer by scanning a different registry.

## File layout

| Thing | Location |
|---|---|
| Script | `global/autoload/<singular>_registry.gd` |
| Resource files | `data/tres/<plural>/` |
| Path constant | `DataPaths.<PLURAL>_DIR` |
| Autoload registration | `project.godot` |
| Audit entry | `global/utils/registry_audit.gd::_check_registry_sizes` |

Examples today: `car_registry.gd` → `data/tres/cars/`, `location_registry.gd` → `data/tres/locations/`.

If the `.tres` directory is generated by tooling and gitignored, say so in a comment next to the `DataPaths` constant — a fresh checkout without the generator run will hit an empty registry and fail `RegistryAudit` at boot, and that comment is what the next person searches for.

## Required API

Every registry exposes, for its resource type `T`:

- `get_<singular>(id: String) -> T` — O(1) lookup. Returns null on miss. This is the one place null is expected.
- `get_all_<plural>() -> Array[T]` — primary iteration API. Returns resources, not ids.
- `get_all_<singular>_ids() -> Array[String]` — kept for serialization boundaries: save dict keys, config reads, debug prints.
- `size() -> int` — used by `RegistryAudit`.

**Iterate resources, not ids.** If the next thing a caller does after `for id in get_all_<singular>_ids()` is `get_<singular>(id)`, they should have iterated resources. The `_ids` form exists for save/load and similar boundaries, not as the default.

## Forbidden APIs

- **No display-name wrappers.** `get_<singular>_display_name(id)` on the registry is banned. Callers take the resource and read `.display_name`. When the id is missing, crashing is correct — that's a data bug, not a fallback condition.
- **No fallback-to-id accessors.** Any helper of the form "return the resource's field X, or the id if missing" hides data bugs. Same reasoning.
- **No item-scanning inverse lookups.** Scanning `ItemRegistry` to answer a category question (or any registry to answer another registry's question) is the anti-pattern this standard exists to prevent. Use a direct reference or an explicit index, below.

**Exception**: query APIs that take an id as a filter parameter are fine. `ItemRegistry.get_items(rarity, category_id)` is a query, not an index return — the id is an input, not a way to look up a single resource.

## Inverse lookups

When registry A needs to answer "which Bs belong to this A?", prefer in order:

1. **Direct reference on the resource.** If `CategoryData` holds `super_category: SuperCategoryData`, answer via `cat.super_category`. Don't build an id-to-id map.
2. **Index built in `_ready`.** If grouping is needed (e.g. "all categories belonging to this super"), build a `Dictionary` at startup after the dependency registry has loaded.
3. **Never by scanning a third registry.** If the shortest path to an answer is "iterate every item and group them", you're rebuilding a map that should have been a direct field.

### Cross-registry index rules

If your `_ready` builds an index by walking another registry:

- Add `assert(OtherRegistry.size() > 0, "<Self> requires <Other> to load first")` at the top of the index build. This catches `project.godot` ordering mistakes at boot instead of letting them manifest as silently-empty indexes.
- Verify autoload order in `project.godot` — dependent registry listed after the one it reads.
- Typed-array values: initialize with `[] as Array[T]`, not bare `[]`. Return `.duplicate()` from the getter so external mutation can't corrupt the index.

## Cross-registry method naming

When one registry exposes a lookup keyed by **another** registry's id, use the `_for` suffix:

```
CategoryRegistry.get_super_category_for(category_id) -> SuperCategoryData
SuperCategoryRegistry.get_super_category(super_category_id) -> SuperCategoryData
```

Both methods would otherwise share the signature `(String) -> SuperCategoryData`, and a caller who typed the wrong receiver would get null at runtime with no type error. The `_for` suffix reads as "look up via another id" and makes the mistake impossible to make silently.

Rule: if a method's parameter id-type differs from the registry's own id-type, the name ends in `_for`.

## Checklist for a new registry

1. Script at `global/autoload/<singular>_registry.gd`, loading via `ResourceDirLoader.load_by_id(DataPaths.<PLURAL>_DIR, id_getter)`.
2. `DataPaths.<PLURAL>_DIR` constant. Comment if gitignored.
3. Autoload line in `project.godot`. Check load order if `_ready` reads other registries.
4. Entry in `RegistryAudit._check_registry_sizes`.
5. Required API only: `get_<singular>`, `get_all_<plural>`, `get_all_<singular>_ids`, `size`. Cross-registry helpers use `_for`. No display-name wrappers, no fallback accessors.
6. If the registry builds an inverse index, add the `assert` and confirm load order.

## PR conventions

When a PR touches a registry, include in the description: "Conforms to `docs/standards/registries.md` §<section>". Reviewers check that section. When this standard changes, the PR that changes it audits existing registries against the new rule and fixes or documents deviations in the same PR.
